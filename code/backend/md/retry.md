好问题。这个机制叫" **自轮询任务** "（self-rescheduling job），它的停止条件和最大重试次数完全取决于代码里写的两个东西：**重新入队的延迟时间**和 **arq 框架的 max_tries 配置** 。

我先帮你重新看一下当前代码，然后讲清楚行为，再给你更稳健的实现。## 当前代码会发生什么

 **当前实现没有任何停止逻辑** 。这是个 bug。我来推演一下会怎样：

每次执行检查"所有 ScoreRun 是否都 COMPLETED"——如果没全完成，就把自己重新入队，5 秒后再跑一次。问题来了：

 **情况 1：评分一切顺利** （最常见）
LLM 大约 5-30 秒一个 task。4 个 task 并行评分，假设最慢 30 秒。`finalise_attempt_if_ready` 第一次在提交后 10 秒触发（`_defer_by=10`），那时候大概 1-2 个 task 评完了，进 if not ready 分支，5 秒后再跑——15 秒、20 秒、25 秒、30 秒、35 秒，第 5 次大概就 ready 了，正常 finalise，循环停止。

所以 **正常情况下大约执行 5-7 次** 。

**情况 2：某个 ScoreRun 永远 FAILED**
这就是 bug 暴露的时候。代码里这一行：

```python
if not any(run.status == ScoreRunStatus.COMPLETED for run in runs):
    ready = False
```

只看有没有 COMPLETED——如果一个 task 的评分失败了，runs 里只有 FAILED 状态，`any(...)` 永远是 False，`ready` 永远是 False。 **这个任务就会每 5 秒重新入队一次，直到天荒地老** 。

**情况 3：LLM API 长时间挂掉**
所有 ScoreRun 都卡在 IN_PROGRESS 或 FAILED——同样无限循环。

## arq 框架本身的兜底

arq 有几个内建机制可以救场，但 **当前代码没有利用它们** ：

 **`max_tries`（默认 5 次）** ：arq 跟踪每个 job 的重试次数，但这只针对**同一个 job_id 因为异常或超时被重试**的情况。我们这里是主动 enqueue 一个新 job——每次都是新的 job_id，所以这个计数对我们没用。

 **`job_timeout`（你设的 300 秒）** ：单次执行超过 5 分钟会被杀掉。但我们的检查只读几次数据库，几毫秒就完了，永远不会触发。

**结论：默认情况下，没有任何东西会让这个自轮询停下来。**

---

## 正确的停止条件应该有四个

一个生产级的"等待并完成"任务应该用**四道闸**控制：## 三个闸各自的作用

 **闸 1（成功路径）** ：所有 task 至少有一个 COMPLETED 状态的 ScoreRun → 跑 finalise_attempt 生成结果，标记 GRADED，结束。

 **闸 2（墙钟超时）** ：从 attempt.submitted_at 算起超过 10 分钟还没成功就放弃。这是兜底中的兜底——给个上限保证最坏情况下学生不会无限期在等待页转圈。

 **闸 3（重试次数）** ：累计入队 N 次还没成功就放弃。这是冗余防护，跟闸 2 互为备份——万一时钟漂移或时区出错导致闸 2 失灵，靠次数也能停下来。

 **闸 4（终态判定）** ：这是 **最关键也最容易漏掉的一条** ——如果所有 ScoreRun 都已经进入终态（COMPLETED 或 FAILED），就不要再等了。比如 4 个 task 中 3 个 COMPLETED、1 个 FAILED，说明这个 FAILED 不会自己变好（worker 不会自动重跑失败的 ScoreRun），继续等是浪费。这时候应该直接进入"部分完成"的处理逻辑——可能是用已完成的部分生成结果（标记某些 task 待人工重评），也可能是直接判定 GRADING_FAILED。

## 数字怎么定

我来推一下默认值：

 **重新入队间隔** ：5 秒。LLM 调用 5-30 秒一次，5 秒粒度足够看到状态变化，又不至于把数据库压垮。

 **成功路径预期次数** ：4 个 task 串行评分最坏 30 秒一个 = 120 秒，前面延迟 10 秒，所以从首次触发到全部完成大约 (120-10)/5 = 22 次。 **实际上 4 个 task 是并行评分的** ，所以瓶颈是最慢的那个，约 30 秒，对应 (30-10)/5 + 1 = 5 次左右。

 **最大重试次数 N** ：取 60 次。60 × 5 秒 = 300 秒 = 5 分钟，比正常 30 秒留了 10 倍 buffer，够应对 LLM 偶发慢。

 **墙钟超时** ：10 分钟。即使 LLM 完全挂了，10 分钟也该让学生看到一个明确的"评分系统暂时不可用"页面，而不是无限等待。

| 1    | 所有 task 都有 COMPLETED →**闸 1 成功**                   | finalise，标记 GRADED，停       |

 **回答你最初的问题** ：

* **正常路径下** ：4 个 task 并行评分约 30 秒，从首次延迟 10 秒触发开始，大约 **执行 5-7 次** 就成功结束。
* **最大检查次数** ： **60 次** （约 5 分钟）。或者距离学生提交超过 10 分钟也会停。两个条件取先到的那个。

## 还有几个值得注意的设计细节

 **为什么用 `attempt_count` 参数而不是用 arq 的 ctx.job_try** ：因为我们每次都 `enqueue_job` 一个新 job——arq 的 try 计数永远是 1。把计数当业务参数传进 payload 是正确的做法。

 **为什么 INVALIDATED 而不是 GRADED** ：失败的 attempt 不应该出现在历史成绩里假装"考过了"。INVALIDATED 状态明确告诉系统"这次不算"，学生需要重考。`client_metadata.grading_failure_reason` 给管理员一个可读的原因，方便排查。

 **为什么不直接重跑失败的 ScoreRun** ：因为 LLM 失败的原因可能是 prompt 触发安全过滤器或者输入超长——这些情况自动重试只会再失败一次。应该走人工评分员 workflow（教师端的"人工评分工作台"页面）来兜底。

 **为什么墙钟超时和重试次数都要有** ：双重保险。如果 worker 进程因为部署或宕机停了好几分钟才恢复，重试次数还远没到 60，但墙钟早就超了——这时候靠闸 2 兜住。如果服务器时间出错（比如 NTP 漂了），靠闸 3 兜住。
