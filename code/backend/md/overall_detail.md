# 一、整个产品的页面地图

按角色分三大块。每块都有一个"中心枢纽页 + 几个具体功能页"的结构。## 三个端各自的页面职责

**学生端的 8 个核心页**：登录、Dashboard、Scenario 列表、Pre-flight 准备页、Scenario Runner（考试容器）、提交后等待页、Result 报告页、历史尝试。再加 2 个辅助页：能力矩阵浏览和个人中心。这是 Demo 必须全部做的。

**教师端的 8 个页面分两条线**。一条是"内容创作"——Scenario 库、Scenario 编辑器、Task 编辑器、Rubric 编辑器，这 4 个页对应你数据库里的 content 域和 rubric 域。另一条是"评分参与"——人工评分工作台、校准报告（AI 跟人工评分对比一致性）、分析报表（题目难度系数、班级分布）。Demo 阶段全部不做，靠 seed 脚本搞定。

**管理员端的 7 个页面**：用户与角色（SSO 集成、班级管理）、能力矩阵版本管理、考试政策（毕业要求设置）、考试场次（formal mode 的开放窗口）、申诉处理、审计日志、系统监控（LLM 用量看板）。Demo 阶段也全部不做。

# 二、场景四涉及的页面深入

场景 4 走完一遍考试涉及 8 个学生端页面，按学生使用顺序我把每个页面要做什么、要包含什么组件、跟后端怎么交互全部讲清楚。## 每个页面的细节

### ① Dashboard 学生主页

**核心目标**：学生进来 3 秒内知道"我现在该做什么"。

**布局**：顶部欢迎语 + 当前 CEFR 等级（如果有过往成绩）；中间一个"开始新尝试"的大 CTA，旁边小字写"Practice 不影响成绩，Formal 计入"；下方是"上次尝试"卡片（如果有）和"推荐 Scenario"。

**API**：`GET /me`、`GET /me/available-scenarios?mode=practice`、`GET /me/attempts?limit=3`。

### ② Scenario 列表

**核心目标**：让学生选场景。Demo 阶段只有 Job Application 一个，但页面要按多场景设计——每个场景是一张卡片。

**每张卡片显示**：场景标题、目标 CEFR 等级（场景 4 是 B2）、估计时长（60 分钟）、覆盖的能力域（Professional + Social-interpersonal）、Practice/Formal 切换器、"开始"按钮。

**关键交互**：Practice 模式不显示倒计时压力提示；Formal 模式上方加红色横幅写"本次成绩计入档案，不可撤回"。

**API**：`GET /scenarios`、`POST /attempts` 创建一次尝试。

### ③ Pre-flight 准备页

这页特别重要，**职业级在线考试都有，但很多新手会跳过**。学生开始考试之前必须确认五件事：

1. 设备检测：浏览器版本（Chrome/Edge 最新）、网络速度、麦克风可用（Task 4 要录音）、扬声器可用（Task 3 要听）。每项一个绿色对勾或红色叉。
2. 考试规则说明：4 个 task、合计 60 分钟、各任务时长、每个任务提交后不可返回修改。
3. 笔记可见性说明：Task 1 的笔记会在 Task 2、Task 4 自动可见；Task 3 的笔记会在 Task 4 可见。
4. 知情同意勾选：录音会被存储用于 AI 评分、可能会用于改进系统、按照 PolyU 数据政策处理。
5. "我已准备好"按钮：勾选完所有同意项才会激活。

**为什么需要这页**：录音和考试时长都是不可逆的资源，学生在 Task 4 才发现麦克风坏了就没法重来了。

**API**：`POST /attempts/{id}/preflight` 记录设备检测结果到 `client_metadata`。

### ④ Scenario Runner（最复杂，单独讲）

这是整个产品最重要的一个页。它是一个**外壳容器**，里面 4 个 Task 用同一个外壳渲染不同的内容。

**外壳 layout（左中右三栏）**：

- **左侧栏（约 240px 宽）**：进度指示——4 个 Task 的圆点 + 标题 + 已用时间，当前任务高亮，已完成的打勾，未到的灰色。下方是"我的笔记"快速预览（Task 1 和 Task 3 的笔记内容，Task 2 和 Task 4 时显示为只读）。底部是个 help 图标。
- **顶部状态栏（约 56px 高）**：场景名 + 当前 Task 名 + **倒计时**（Task 时间，最后 2 分钟变红）+ **自动保存指示**（"已保存 5 秒前"或 spinner）+ 网络状态点。
- **主内容区**：根据 `task_type` 动态渲染不同的子组件。
- **底部操作栏（约 64px 高）**：左侧是"上一题"（如果允许，场景 4 不允许所以隐藏）；右侧是"提交本任务并继续 →"按钮。提交后弹一个二次确认。

**4 个 Task 主内容区分别长这样**：

**Task 1（阅读 + 笔记）布局**：左半屏是 BrightWave 公司广告全文，可滚动可高亮（高亮内容前端记录，不发后端）。右半屏是笔记区，分两个分组——上面"3-5 个 qualities"输入区（动态加减条目，每条一行短语），下面"3 个 responsibilities"输入区，再下面是"选择你要申请的职位"单选三个选项。计数显示"已写 4/5"。

**Task 2（写求职信）布局**：左半屏是只读的"参考资料"标签——Tab 1 是 Task 1 的笔记，Tab 2 是 Job Ad 全文（可折叠）。右半屏是富文本编辑器（最简版即可，支持段落/换行就行），下面有字数统计"287 / 250-300 words"，超出范围标红。

**Task 3（听讲座 + 笔记）布局**：上半屏是音频播放器组件——播放/暂停/进度条/速度切换（0.75x/1.0x/1.25x）。**关键细节：要不要允许暂停和回放，是个产品决策**——B2 听力的真实测试一般只能听 1-2 遍。建议显示"剩余可重听次数 1/1"。下半屏是笔记区，结构同 Task 1（vision 分组 + additional qualities 分组）。

**Task 4（面试录音）布局**：上半屏显示当前问题（4 个问题串行）+ 上一问的回答状态。中间一个**录音组件**：大圆形录音按钮、波形可视化、当前录音时长、最大 3 分钟提示。下半屏左侧显示"参考你的笔记"折叠面板（Task 1 + Task 3 的笔记可见）。

**录音的产品细节**：浏览器 MediaRecorder API 录 webm 格式 → 学生点"完成本问"→ 前端先 POST `/artifacts` 拿到预签名 URL → PUT 到对象存储 → PATCH 提交 checksum。整个过程 UI 显示三档进度："上传中... 70%"。**强烈建议本地 IndexedDB 同时存一份**，万一上传失败可以重试。

**核心交互逻辑**：

- 每 30 秒自动 PUT `/attempts/{aid}/tasks/{tid}/response`（除了 Task 4 录音）
- 倒计时到 0 自动提交当前 task 并跳下一个
- 用户切到别的浏览器 tab 触发 proctoring 事件（POST 到后端）
- 网络断开时本地暂存，恢复后批量重发

### ⑤ 提交确认页

**别小看这页**。学生做完 Task 4 之后跳到这里，显示一个清单：

```
✓ Task 1 已完成（4 个 qualities, 3 个 responsibilities）
✓ Task 2 已完成（287 字）
✓ Task 3 已完成（3 个 vision, 2 个 qualities）
✓ Task 4 已完成（4 段录音，总长 6:42）

[ 我确认提交 ]    [ 返回检查 ]
```

**为什么要有这页**：让学生最后看一眼，避免误提交。点了确认就调 `POST /attempts/{id}:submit`，attempt 变成 GRADING 状态，不可逆。

### ⑥ 评分等待页

**核心目标**：让学生知道"系统正在干活，没死机"。

**布局**：一个进度条显示 4 个 task 的评分进度（每个 task 一行）：

```
Task 1 笔记       ✓ 已评分
Task 2 求职信     ⟳ 评分中（LLM 调用）
Task 3 听力笔记   ⟳ 排队中
Task 4 面试       ⟳ 转写中（ASR 处理）
```

**实现**：前端每 3 秒轮询 `GET /attempts/{id}/result`——返回 409 则刷新进度条，返回 200 则跳到 Result 页。或者用 SSE/WebSocket 推送，但 MVP 阶段轮询足够。

**预期等待时间**：Demo 用 mock 几秒就好，真实 LLM 评分预计 1-3 分钟。要在页面上明确告诉学生"通常 2-3 分钟"，避免他们以为卡死。

### ⑦ Result 报告页

**这是整个产品的高光时刻**。学生最在乎这页。

**布局自上而下**：

1. **顶部 Hero 区**：大字显示总体 CEFR 等级（如 "B2"）+ 一句话评语（"You meet the B2 standard for graduation in this scenario."）。Pass/Fail 模式下还要显示 ✓/✗ 大图标。
2. **能力画像可视化**：一个 5×3 的雷达图或表格，5 个域（行）× 3 个 competence（列），每个单元格用颜色块表示等级（A2 灰、B1 黄、B2 绿、C1 蓝）。这就是你后端 `attempt_results.competency_profile` 字段渲染出来的。
3. **任务级详情**（可折叠的 4 个面板）：每个 task 一个折叠面板，展开后显示：该任务的总体 band、各 criterion 的 band 评分、AI 给出的简短理由（rationale）、对应作答片段引用（evidence_refs）。
4. **改进建议**：基于得分较低的 criterion 自动生成建议。这个 Demo 阶段可以直接用规则模板，比如"linguistic 拿了 band 2 → '尝试在写作中使用更多复杂句和连接词'"。
5. **操作按钮**：下载 PDF 报告（后续）、再练一次（创建 practice 模式新 attempt）、提出申诉（如果是 formal 模式且仍在申诉期内）。

**API**：`GET /attempts/{id}/result`，返回的就是 `AttemptResultRead` 这个 schema。

### ⑧ 历史尝试

简单的列表页。每行一个 attempt：日期、scenario 名、mode、CEFR 等级、查看详情链接。

# 三、系统交互图（mermaid）

下面是学生完整考一次的端到端时序图，把前端、后端 API、对象存储、Redis 队列、LLM 之间的所有调用全部画出来。让我用 mermaid sequence diagram 渲染时序图。这个比 SVG 更适合，因为节点和消息交互很多。## 几个时序图里特别需要注意的细节

**Phase 2 那个循环里的 PUT 请求是幂等的**——这就是为什么我前面把"自动保存"设计成 PUT 而不是 POST。学生网络抖动重发同样的请求，数据库不会出问题。

**Phase 3 录音上传走的是预签名 URL 双跳**。学生浏览器先跟 API 要一个临时 URL，然后直接把音频字节 PUT 到对象存储，最后再回头通知 API 写入元数据。**音频字节从来不经过你的 API 服务器**——这是性能和成本上的关键设计。10MB 的录音如果走 API，光带宽费就够呛。

**Phase 5 是并行的**——4 个 task 的评分任务同时进队列，Worker 池子可以同时跑 4 个 LLM 调用。Task 4 多了一步 ASR 转写所以慢一些，但前 3 个不用等它。

**Phase 6 的"汇总"是触发式而不是周期性**。`finalise_attempt_if_ready` 这个 worker 任务会检查所有 ScoreRun 都完成了没——没完成就把自己重新入队延迟 5 秒再试，完成了就生成 AttemptResult。这避免了用 cron 轮询数据库。

**Phase 7 学生轮询返回 409 而不是空 200**——409 Conflict 在 HTTP 语义上正好表示"资源还没准备好"，前端拿到 409 就继续轮询，拿到 200 就跳页。这比返回 `{"status": "pending"}` 200 更符合 REST 语义。

## 一个产品决策需要你拍板

**Task 3 听讲座要不要允许多次重听？** 这个决定影响 UI 设计也影响测量效度：

- **允许无限重听** → 学生体验好，但测的更多是"理解 + 笔记技巧"而不是真实听力。
- **只能听 1 次** → 跟传统听力考试一致，但学生压力大、网络抖动可能毁掉整道题。
- **可以听 2 次但要记录次数** → 折中方案，给到学生缓冲，AI 评分时把"用了几次重听"作为一个补充信号。

我的建议是**第三种**。如果你同意我可以把这个状态加到 `task_responses.derived_analysis` 的 schema 里，并且在 Scenario Runner 的 Task 3 子组件里加上"剩余重听次数 1/2"的 UI 元素。
